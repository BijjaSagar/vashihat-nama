1. Data Security on the Device (Client-Side)
Apple’s security starts with the Secure Enclave (SEP), a dedicated hardware chip isolated from the main processor.

Hardware Root of Trust: Your encryption keys are never stored in the main software. They are stored inside the Secure Enclave. Even if a hacker gets full access to your iPhone’s operating system (iOS), they cannot "read" the keys out of the SEP.
File-Level Encryption: Apple uses APFS (Apple File System). Every file on your device is encrypted with its own unique key. These keys are then "wrapped" (encrypted) by a class key, which is tied to your passcode.
Data Protection API: iOS uses 4 protection classes (e.g., Complete Protection, Protected Until First User Authentication). This ensures that if the phone is locked, the keys to decrypt sensitive files are actually purged from memory.

2. Data Storage in the Cloud (iCloud)
When data moves to iCloud, Apple uses a complex "Chunking" and "Encryption" process:

Chunking: Large files (like videos) are broken into small chunks.
Encryption at Rest: Each chunk is encrypted using AES-256 (Advanced Encryption Standard).
Storage Providers: Apple actually stores the encrypted chunks on third-party servers like AWS S3 and Google Cloud, but those providers never have the keys. Only Apple (or you) has the keys to reassemble and decrypt the chunks.

3. Data Encryption (The Algorithms)
Symmetric Encryption: Apple uses AES-256-GCM (Galois/Counter Mode). This is extremely fast on modern CPUs and provides both encryption and "integrity" (proving the data hasn't been tampered with).
Asymmetric Encryption: For sharing data and key management, they use Elliptic Curve Cryptography (ECC) (specifically NIST P-256 or Curve25519). This is used to "wrap" the AES keys so they can be sent securely between devices.
Advanced Data Protection (ADP): This is Apple’s "Zero-Knowledge" mode. When enabled, Apple deletes the keys from their servers, and the keys exist only on your trusted devices. If you lose your devices and your recovery key, Apple cannot recover your data—exactly the "Zero-Knowledge" model you are aiming for in your project.

4. Data Compression (Performance & Cost)
To save storage space in iCloud and speed up sync, Apple uses a custom algorithm they developed:

LZFSE (Lempel-Ziv Finite State Entropy): This is Apple's primary compression algorithm.
Why they use it: It provides the high compression ratio of ZLIB (standard ZIP) but is 3x faster and uses much less battery power.
Logic: It uses "Finite State Entropy," a modern mathematical way to pack data tighter than old Huffman coding used in standard ZIP files.
LZ4: Used for scenarios where "extreme speed" is needed (like real-time system tasks), though the files are slightly larger than LZFSE.
ProRes & HEVC: For "Video Wills" or video files, Apple uses hardware-accelerated video codecs (H.265/HEVC) which can shrink video size by 50% compared to standard H.264 without losing quality.
